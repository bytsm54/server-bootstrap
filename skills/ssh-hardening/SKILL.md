# SSH Hardening

One-command SSH security hardening. Compatible with VS Code Remote SSH, Cursor, and other IDE remote development tools.

## Parameters

Ask the user for required parameters if not provided:

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `ssh_port` | yes | — | SSH port number (e.g., `22022`) |
| `allow_users` | yes | — | Space-separated list of allowed SSH users (e.g., `ubuntu`) |
| `ide_compatible` | no | `true` | Enable TCP/agent/stream forwarding for IDE Remote SSH |
| `fail2ban_bantime` | no | `3600` | fail2ban ban duration in seconds |
| `fail2ban_maxretry` | no | `3` | Max failed attempts before ban |

## Execution

Follow these steps in exact order. The safety checks are critical — do not skip them.

### Step 1: Backup Existing Configuration

```bash
sudo mkdir -p /tmp/sshd_backup
sudo cp /etc/ssh/sshd_config /tmp/sshd_backup/
sudo cp /etc/ssh/sshd_config.d/*.conf /tmp/sshd_backup/ 2>/dev/null
```

### Step 2: Write Hardening Configuration

Write to `/etc/ssh/sshd_config.d/99-hardening.conf`:

```
# SSH Hardening Configuration — generated by ssh-hardening skill
Port <ssh_port>

# --- Authentication ---
PermitRootLogin no
PasswordAuthentication no
PermitEmptyPasswords no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
MaxAuthTries 3
MaxSessions 5
LoginGraceTime 30
AllowUsers <allow_users>

# --- Keep-alive (IDE Remote SSH compatible) ---
ClientAliveInterval 60
ClientAliveCountMax 3
TCPKeepAlive yes

# --- Forwarding ---
AllowTcpForwarding <yes if ide_compatible else no>
AllowStreamLocalForwarding <yes if ide_compatible else no>
AllowAgentForwarding <yes if ide_compatible else no>
X11Forwarding no
GatewayPorts no

# --- Strong ciphers ---
Ciphers aes256-gcm@openssh.com,chacha20-poly1305@openssh.com,aes256-ctr
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com
KexAlgorithms curve25519-sha256,curve25519-sha256@libssh.org,ecdh-sha2-nistp521

# --- Security ---
UseDNS no
LogLevel VERBOSE
```

Replace `<ssh_port>`, `<allow_users>`, and forwarding values based on user parameters.

### Step 3: Validate Configuration

This is the safety gate. Run:

```bash
sudo sshd -t
```

**If `sshd -t` fails:** immediately restore the backup and stop. Do not proceed to restart sshd.

```bash
sudo cp /tmp/sshd_backup/sshd_config /etc/ssh/sshd_config
sudo rm -f /etc/ssh/sshd_config.d/99-hardening.conf
```

Report the error to the user and do not continue.

**If `sshd -t` succeeds:** proceed to the next step.

### Step 4: Configure fail2ban

Install if not present:

```bash
sudo apt-get install -y fail2ban
```

Write `/etc/fail2ban/jail.local`:

```
[sshd]
enabled = true
port = <ssh_port>
maxretry = <fail2ban_maxretry>
bantime = <fail2ban_bantime>
findtime = 600
```

Restart fail2ban:

```bash
sudo systemctl restart fail2ban
sudo systemctl enable fail2ban
```

Verify: `sudo fail2ban-client status sshd` shows the jail is active.

### Step 5: Restart sshd

Restarting sshd does NOT kill existing connections — the current session is safe.

```bash
sudo systemctl restart sshd
```

Verify sshd is listening on the new port:

```bash
sudo ss -tlnp | grep <ssh_port>
```

### Step 6: Safety Reminders

After completing all steps, remind the user:

1. **Firewall / Security Group**: Open port `<ssh_port>` in your cloud provider's security group or firewall rules before closing this session
2. **Test before disconnecting**: Open a NEW terminal and test connecting on the new port before closing the current session:
   ```
   ssh -p <ssh_port> <user>@<server-ip>
   ```
3. **Update local SSH config**: Update `~/.ssh/config` on your local machine with the new port

## Rollback

If anything goes wrong after restart, the backup is at `/tmp/sshd_backup/`. Restore with:

```bash
sudo cp /tmp/sshd_backup/* /etc/ssh/ /etc/ssh/sshd_config.d/
sudo rm -f /etc/ssh/sshd_config.d/99-hardening.conf
sudo systemctl restart sshd
```
